## 🟢 Баг 1 (лёгкий) — Обменник: нулевая сумма обмена

**Файл:** `lib/ui/screens/exchange_screen.dart`
**Метод:** `_performExchange()`, строка 213

### Проблемный код

```dart
if (amount == null || amount < 0) {
  ScaffoldMessenger.of(context).showSnackBar(
    const SnackBar(content: Text('Введите корректную сумму')),
  );
  return;
}
```

### Суть

Валидация проверяет `amount < 0`, но **не** `amount == 0`. Пользователь может ввести `0` в поле суммы, нажать «Обменять», и в историю сохранится операция `0 RUB → 0 USD` с нулевыми значениями.

### Шаги воспроизведения

1. Открыть экран «Обменник»
2. Дождаться загрузки курсов
3. В поле «Сумма» ввести `0`
4. Нажать кнопку «Обменять»

### Ожидаемое поведение

Приложение показывает ошибку «Введите корректную сумму» и не создаёт операцию.

### Фактическое поведение

Операция `0.00 RUB → 0.00 USD` сохраняется в историю обменов.

### Как исправить

Заменить `amount < 0` на `amount <= 0`:

```dart
if (amount == null || amount <= 0) {
```

## 🟢 Баг 2 (лёгкий) — Настройки: «Очистить все данные» очищает не всё

**Файл:** `lib/state/app_state.dart`
**Метод:** `clearAll()`, строки 75–79

### Проблемный код

```dart
Future<void> clearAll() async {
  _items.clear();
  notifyListeners();
  await _repo.clear();
}
```

### Суть

Метод `clearAll()` очищает только `_items` (записи расходов/доходов) через `_repo.clear()`. При этом **не очищаются**:
- `exchange_operations` — история обменов валют
- `portfolio_holdings` — позиции в портфеле
- `portfolio_transactions` — история сделок портфеля
- баланс портфеля

Кнопка в настройках называется «Очистить все данные», что вводит пользователя в заблуждение.

### Шаги воспроизведения

1. Создать несколько расходов/доходов
2. Выполнить обмен валюты на экране «Обменник»
3. Купить валюту в «Портфеле»
4. Перейти в «Настройки» → «Очистить все данные» → подтвердить
5. Проверить экраны «Обменник» и «Портфель»

### Ожидаемое поведение

Все данные приложения удалены: расходы, история обменов, позиции и сделки портфеля.

### Фактическое поведение

Расходы/доходы удалены, но история обменов и данные портфеля остались на месте.

### Как исправить

Добавить в `clearAll()` очистку остальных репозиториев:

```dart
Future<void> clearAll() async {
  _items.clear();
  notifyListeners();
  await _repo.clear();
  await ExchangeRepository().clear();
  await PortfolioRepository().clearAll();
}
```

Либо — изменить текст кнопки на «Очистить только расходы и доходы».

## 🟡 Баг 3 (средний) — Портфель: деление на ноль при нулевом курсе

**Файл:** `lib/ui/screens/portfolio_screen.dart`
**Метод:** `_rateToBase()`, строки 69–74

### Проблемный код

```dart
double? _rateToBase(String currency) {
  if (_rates == null) return null;
  if (currency == 'RUB') return 1.0;
  final r = _rates!.rates[currency];
  return r != null ? 1 / r : null;
}
```

### Суть

Метод возвращает `1 / r`, но **не проверяет**, что `r > 0`. Если API (или мок/кэш) вернёт курс `0.0` для какой-либо валюты, произойдёт деление на ноль. В Dart `1 / 0.0` даёт `Infinity` (тип `double`), что приведёт к:
- Отображению `Infinity` или `NaN` в UI
- Некорректным расчётам стоимости портфеля, PnL, аллокации
- Потенциально — к созданию операций с бесконечными суммами

### Шаги воспроизведения

1. Симулировать ответ API с курсом `0.0` для одной из валют (например, через мок или подмену `rates_api.dart`)
2. Открыть экран «Портфель»
3. Наблюдать значения `Infinity` в блоке баланса и кнопках покупки

> 💡 Без подмены API воспроизвести сложно, но баг реален — курсы приходят от внешнего сервиса, и значение `0` не исключено.

### Ожидаемое поведение

При нулевом курсе метод возвращает `null`, и интерфейс обрабатывает отсутствие курса корректно (как для неизвестной валюты).

### Фактическое поведение

Метод возвращает `Infinity`, что ломает расчёты и отображение.

### Как исправить

Добавить проверку `r > 0` перед делением:

```dart
return r != null && r > 0 ? 1 / r : null;
```

## 🟡 Баг 4 (средний) — Запись расхода/дохода с нулевой суммой

**Файл:** `lib/ui/screens/add_edit_screen.dart`
**Метод:** валидатор поля «Сумма» в `build()`, строка 194

### Проблемный код

```dart
validator: (v) {
  final x = double.tryParse((v ?? '').replaceAll(',', '.'));
  if (x == null || x < 0) return 'Введите неотрицательную сумму';
  return null;
},
```

### Суть

Валидатор проверяет `x < 0`, но **не** `x == 0`. Пользователь может сохранить запись расхода или дохода с суммой `0`, которая:
- Отображается в списке как запись с нулевой суммой
- Искажает статистику (сумма по категории, средние и т.п.)
- Не имеет практического смысла

### Шаги воспроизведения

1. Нажать «+» (добавить запись)
2. Ввести название — любое (например, «Тест»)
3. В поле «Сумма» ввести `0`
4. Нажать «Сохранить»

### Ожидаемое поведение

Валидатор показывает ошибку, запись не сохраняется.

### Фактическое поведение

Запись с суммой `0.00` успешно сохраняется и появляется в списке.

### Как исправить

Заменить `x < 0` на `x <= 0` и обновить текст ошибки:

```dart
if (x == null || x <= 0) return 'Введите сумму больше 0';
```

## 🔴 Баг 5 (сложный) — Обменник: `setState` после `dispose`

**Файл:** `lib/ui/screens/exchange_screen.dart`
**Метод:** `_loadRates()`, блок `catch`, строка 96

### Проблемный код

```dart
Future<void> _loadRates() async {
  setState(() {                    // ← тоже без mounted, но менее критично (синхронный)
    _rates = null;
    _ratesError = null;
  });
  try {
    final r = await RatesApi.fetch();
    if (mounted) {                 // ✅ проверка mounted есть
      setState(() => _rates = r);
      // ...
    }
  } catch (e) {
    debugPrint('[FinControl] ExchangeScreen: ошибка загрузки курсов — $e');
    setState(() => _ratesError = e is Exception ? e : Exception(e.toString()));
    //     ↑ ❌ НЕТ проверки mounted!
  }
}
```

### Суть

В блоке `try` перед `setState` есть проверка `if (mounted)` — корректно. Но в блоке `catch` вызов `setState(() => _ratesError = ...)` выполняется **без проверки `mounted`**.

Если пользователь быстро уходит с экрана «Обменник» во время загрузки курсов, и при этом запрос завершается с ошибкой (таймаут, нет сети), виджет уже будет `dispose`'d, и вызов `setState` вызовет исключение:

```
FlutterError: setState() called after dispose()
```

### Шаги воспроизведения

1. Отключить интернет (или включить режим «В самолёте»)
2. Открыть экран «Обменник» — начнётся загрузка курсов
3. **Быстро** переключиться на другой экран (например, «Расходы») до завершения загрузки
4. Дождаться таймаута сетевого запроса

> 💡 В debug-сборке ошибка `setState() called after dispose()` появится в консоли. В release-сборке может привести к крэшу или молчаливому сбою.

### Ожидаемое поведение

Ошибка сети обрабатывается тихо, без вызова `setState` на уже уничтоженном виджете.

### Фактическое поведение

Вызывается `setState` после `dispose`, что приводит к ошибке Flutter.

### Как исправить

Обернуть `setState` в блоке `catch` проверкой `mounted`:

```dart
} catch (e) {
  debugPrint('[FinControl] ExchangeScreen: ошибка загрузки курсов — $e');
  if (mounted) {
    setState(() => _ratesError = e is Exception ? e : Exception(e.toString()));
  }
}
```

## 📝 Заметки для экзаменатора

- **Баги 1 и 4** — проверка граничных значений (boundary value testing). Ожидается, что ученик найдёт хотя бы один из них.
- **Баг 2** — функциональное тестирование кнопки «Очистить все данные». Требует проверки всех экранов после действия.
- **Баг 3** — требует понимания работы с API и edge-case значений. Сложно воспроизвести без мока, но ученик с опытом должен увидеть отсутствие защиты от деления на ноль при code review.
- **Баг 5** — lifecycle-баг Flutter (`mounted`). Требует понимания асинхронности и жизненного цикла `State`. Воспроизведение требует определённого тайминга (быстрый уход с экрана + ошибка сети).
