# Практика: Firebase Performance Monitoring — регресс по скорости

**Одно приложение** FinControl. **Сначала [00-firebase-setup.md](00-firebase-setup.md):** свой проект Firebase, конфиги в проект, `Firebase.initializeApp()`. Затем подключаешь Performance, добавляешь кастомный трейс (например загрузка курсов) и смотришь метрики в консоли — регрессии по скорости между версиями.

## Цель

Подключить Firebase Performance Monitoring к приложению FinControl (твой проект Firebase), собрать автоматические трейсы (запуск приложения, сетевые запросы) и кастомный трейс загрузки курсов, смотреть метрики в консоли и сравнивать версии на предмет регрессий по скорости.

## Важно: один проект Firebase

**Сначала [00-firebase-setup.md](00-firebase-setup.md):** свой проект, свои конфиги, `Firebase.initializeApp()`. Ключи Sentry/AppMetrica (практики 06–07) — в [STUDENT_ENV.md](../STUDENT_ENV.md), не здесь.

## Ожидаемый результат

- Performance Monitoring включён; автоматические метрики (App start и при настроенном перехвате — сетевые запросы) и кастомный трейс (например `load_rates`) собираются.
- В Firebase Console → **Performance** → Dashboard и **Traces** видны метрики и трейсы; по ним можно оценить время ключевых операций и сравнить сборки на регрессии.

## Что понадобится

- Ваш Firebase-проект с включённым Performance (в консоли)
- Добавьте в `pubspec.yaml` при необходимости `firebase_performance` (совместимую с firebase_core)

## Шаг 1: Подключение

Добавьте при необходимости `firebase_performance` в `pubspec.yaml`, выполните `flutter pub get`. После `Firebase.initializeApp()` (по [00-firebase-setup.md](00-firebase-setup.md)) Performance обычно начинает автоматически собирать:

- **App start** — время до первого кадра.
- **HTTP-запросы** — если использовать перехват (см. документацию пакета для Flutter).

Для кастомных операций создайте трейс:

```dart
import 'package:firebase_performance/firebase_performance.dart';

final trace = FirebasePerformance.instance.newTrace('load_rates');
await trace.start();
try {
  await RatesApi.fetch();
} finally {
  await trace.stop();
}
```

Добавьте такой трейс вокруг загрузки курсов (в экране обменника или главном экране), чтобы видеть время загрузки курсов в отчётах.

## Шаг 2: Просмотр в Firebase Console

1. **Performance** → **Dashboard**: обзор по метрикам (время запуска, время до интерактивности и т.д.).
2. **Traces** — список кастомных и автоматических трейсов; можно сравнить по версиям приложения и устройствам.
3. **Network** — если настроен перехват HTTP, видны запросы к API курсов и их длительность.

## Шаг 3: Поиск регрессий

- Сравните метрики между версиями (например до и после изменения кода).
- Фильтруйте по устройству/ОС — регресс может проявляться только на части устройств.
- Используйте кастомный трейс `load_rates`, чтобы отслеживать регресс скорости загрузки курсов.

## Рекомендуемые метрики и трейсы

| Метрика/трейс | Описание | Где смотреть |
|---------------|----------|--------------|
| **App start** | Время до первого кадра (автоматически) | Performance → Dashboard, Traces |
| **load_rates** | Время загрузки курсов (сеть или кэш) — ключевой сценарий | Traces → кастомный трейс |
| **Network** | Длительность HTTP-запросов к API курсов | Performance → Network (при настроенном перехвате) |

Дополнительно при необходимости можно добавить кастомные трейсы: открытие экрана обменника/портфеля, сохранение расхода в БД, загрузка списка акций (`StocksApi.fetch`) — по тому же шаблону `newTrace('имя')` → `start()` → операция → `stop()`.

### Как искать регрессии

1. **Фиксация базовой версии:** после стабильной сборки посмотри в Firebase Console → Performance → Traces среднюю длительность `load_rates` и App start для выбранного типа устройств (например, только Android).
2. **После изменений:** сделай новую сборку, установи на то же или похожее устройство, повтори сценарий (открытие приложения, переход на экран с курсами). Дождись появления данных в Performance (задержка до нескольких часов).
3. **Сравнение:** в Traces выбери трейс `load_rates` (и при необходимости App start), отфильтруй по версии приложения. Сравни метрики «до» и «после». Рост длительности при тех же условиях сети указывает на возможную регрессию.
4. **Учёт сети:** регрессия — рост времени при сопоставимой сети. Если у пользователей медленная сеть, смотри сегмент по скорости сети в отчёте или сравнивай только схожие условия.
5. **Что менять при регрессии:** проверь последние правки в `RatesApi`, экранах обменника/главной, в коде инициализации БД и загрузки кэша — не появились ли синхронные тяжёлые операции на main isolate.
