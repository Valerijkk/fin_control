# 🍎 Практика 04: Xcode — симулятор iOS, логирование и инструменты тестирования

> Открываем проект FinControl в Xcode, запускаем на симуляторе iOS, настраиваем просмотр логов в **Console** и осваиваем инструменты Xcode для тестирования: Instruments, View Debugger, Memory Graph, а также функции симулятора (VoiceOver, размеры экранов, запись видео).

---

## 🎯 Цель

Открыть проект FinControl в Xcode (`ios/Runner.xcworkspace`), запустить приложение в симуляторе iOS, настроить просмотр логов в Console и освоить встроенные инструменты Xcode для тестирования производительности, UI и accessibility.

---

## ✅ Ожидаемый результат

- ✔ В Xcode открыт **`ios/Runner.xcworkspace`** (не `.xcodeproj`!), выбрана схема **Runner** и симулятор (например iPhone 15, iOS 17)
- ✔ Приложение FinControl успешно собирается и запускается в симуляторе; на экране отображается приложение с навигацией
- ✔ В нижней панели Xcode (Debug area) открыта вкладка **Console**; при действиях в приложении в консоли появляются логи (`print`, `debugPrint`, системные сообщения)
- ✔ Освоены базовые инструменты: Instruments, View Debugger, возможности симулятора

|-----|-------|
| **macOS** с установленным **Xcode** | Среда разработки и симулятор iOS |

## 📝 Пошаговая инструкция

### Шаг 1: Открытие проекта в Xcode

1. Убедись, что в корне проекта FinControl (где `pubspec.yaml`) есть папка `ios/`.
2. Открой **именно** файл **`ios/Runner.xcworkspace`** — **не** `Runner.xcodeproj`.

   > ⚠️ Открывать нужно `.xcworkspace`, а не `.xcodeproj` — иначе CocoaPods-зависимости и Flutter-плагины не подтянутся, и сборка упадёт с ошибками.

3. Открой двойным кликом или из терминала в корне проекта:
   ```bash
   open ios/Runner.xcworkspace
   ```
4. Дождись загрузки проекта в Xcode (при первом открытии может занять минуту-две для индексации).

---

### Шаг 2: Выбор симулятора и запуск

1. В Xcode в верхней панели слева найди выпадающий список рядом с кнопкой **Run** (▶).
2. Выбери схему **Runner**.
3. В том же выпадающем списке выбери **симулятор** (например **iPhone 15**, iOS 17).

   > ⚠️ Не выбирай «Any iOS Device» — это режим сборки для реального устройства, а не для симулятора.

4. Нажми **Run** (▶) или сочетание **Cmd + R**.
5. Дождись окончания сборки. При первой сборке Flutter и CocoaPods подтянут зависимости — это может занять несколько минут.
6. Приложение установится и запустится в симуляторе; откроется окно симулятора с FinControl.

#### Альтернатива: запуск из терминала

Из корня проекта:

```bash
# Проверь доступные устройства
flutter devices

# Запусти на iOS-симуляторе
flutter run
```

Выбери iOS-симулятор из списка, если устройств несколько.

---

### Шаг 3: Просмотр логов в Xcode (Console)

1. Во время работы приложения открой нижнюю панель (Debug area):
   - Меню **View → Debug Area → Activate Console**, или
   - Горячие клавиши **Cmd + Shift + C**
2. Убедись, что активна вкладка **Console** (внизу панели).
3. В консоли отображаются:
   - Вывод `print()` и `debugPrint()` из Dart-кода
   - Логи нативного кода iOS
   - Системные сообщения Flutter-движка
4. В приложении выполни действия: открой экран **Обменник**, **Портфель**, добавь расход — в консоли должны появляться новые строки.

---

### Шаг 4: Фильтрация логов

1. В **поле поиска** консоли (внизу Debug area) введи ключевое слово для фильтрации:
   - `[FinControl]` — логи приложения с тегом
   - Название экрана (например `Exchange`, `Portfolio`)
   - `Error` или `Exception` — для поиска ошибок
2. Для просмотра только ошибок ориентируйся на уровни вывода и стек-трейсы.

> 💡 Для расширенного анализа логов можно использовать системное приложение **Console.app** (Applications → Utilities → Console.app). В левой панели выбери симулятор, в поиске введи `FinControl` или `flutter` — увидишь все системные логи, включая нативные сообщения iOS.

## 🛠 Траблшутинг

**Ошибки подписания / Signing**
→ Открой таргет Runner → **Signing & Capabilities**, включи **Automatically manage signing** и выбери свою команду (Apple ID). Для симулятора достаточно бесплатного аккаунта.

**Сборка не находит Flutter**
→ Выполни из корня проекта `flutter pub get`, затем открой снова `ios/Runner.xcworkspace`.

**Ошибки CocoaPods**
→ Выполни из корня проекта:
```bash
cd ios && pod install && cd ..
```
Затем открой `Runner.xcworkspace` заново.

**Симулятор не появляется в списке**
→ Открой **Xcode → Settings → Platforms** и убедись, что скачан нужный iOS runtime. При необходимости нажми **+** и загрузи.

**Долгая первая сборка**
→ Это нормально. Flutter и CocoaPods скачивают зависимости. Последующие сборки будут быстрее.

## 🧪 Практические сценарии

### Сценарий 1: Отслеживание логов при навигации

1. Запусти FinControl в симуляторе через Xcode (**Cmd + R**).
2. Открой Console (**Cmd + Shift + C**).
3. В строке поиска введи `[FinControl]`.
4. В приложении нажми **Начать** → в Console:
   ```
   [FinControl] Запуск приложения...
   [FinControl] Данные загружены: N записей, M категорий
   ```
5. Перейди на **Обменник** → **Портфель** → **Акции**:
   ```
   [FinControl] ExchangeScreen: курсы загружены...
   [FinControl] PortfolioScreen: загружено — баланс...
   [FinControl] StocksScreen: загружено N акций
   ```

---

### Сценарий 2: Тестирование на разных устройствах

1. Запусти FinControl на **iPhone SE** (маленький экран).
2. Проверь: не обрезается ли текст? Не вылезают ли кнопки за экран?
3. Переключи на **iPhone 15 Pro Max** (большой экран).
4. Проверь: нет ли слишком много пустого пространства?
5. Переключи на **iPad** (если доступно).
6. **Запиши:** на каких устройствах UI выглядит хорошо, на каких есть проблемы.

> 💡 Переключение между симуляторами: в Xcode измени устройство в верхней панели и нажми **Cmd + R** заново.

---

### Сценарий 3: Проверка Accessibility (VoiceOver)

1. В симуляторе: **Settings → Accessibility → VoiceOver** → включи.
2. Тапай по элементам FinControl и слушай, что озвучивает VoiceOver.
3. **Что проверить:**
   - Кнопки озвучиваются? («Обменять», «Купить», «Добавить»)
   - Суммы читаются? («100 000 рублей»)
   - Графики (свечные) имеют описание? (скорее всего нет — это баг accessibility)
4. Выключи VoiceOver после теста.

> ⚠️ С включённым VoiceOver управление меняется: одиночный тап выделяет элемент, двойной тап активирует. Для навигации используй свайпы влево/вправо.

---

### Сценарий 4: Тестирование при увеличенном шрифте

1. В симуляторе: **Settings → Accessibility → Display & Text Size → Larger Text** → установи максимум.
2. Открой FinControl.
3. **Проверь:** не ломается ли вёрстка? Читается ли текст? Не наезжают ли элементы друг на друга?
4. Это важный тест для реальных приложений — многие пользователи увеличивают шрифт.

## 🔧 Дополнительные инструменты Xcode

### Instruments — профилирование производительности

Instruments — мощный инструмент профилирования в Xcode.

1. **Product → Profile** (**Cmd + I**) — собирает приложение и открывает Instruments.
2. Выбери шаблон:

   | Шаблон | Что анализирует |
   |--------|----------------|
   | **Time Profiler** | Какие методы занимают больше всего CPU |
   | **Allocations** | Использование памяти и утечки |
   | **Leaks** | Обнаружение утечек памяти |
   | **Network** | Сетевая активность |

3. Нажми **Record** (красная кнопка), выполни сценарий в приложении, нажми **Stop**.
4. Анализируй результаты: ищи горячие точки (hot spots) в коде.

> 💡 **Сценарий:** профилируй загрузку экрана Обменник — проверь, нет ли блокирующих операций на main thread.

---

### View Debugger — 3D-иерархия UI

1. Запусти приложение в симуляторе через Xcode (**Cmd + R**).
2. **Debug → View Debugging → Capture View Hierarchy** (или кнопка в Debug bar).
3. Откроется 3D-визуализация всех слоёв интерфейса.
4. Кликай на элементы — в инспекторе справа увидишь свойства (frame, constraints, цвет, текст).

**Что проверять:**
- Нет ли невидимых перекрывающих view
- Правильны ли размеры и позиции элементов
- Нет ли лишних слоёв (влияет на производительность рендеринга)

---

### Memory Graph Debugger — утечки памяти

1. Запусти приложение, походи по экранам.
2. В Debug bar нажми **Memory Graph** (иконка с узлами).
3. Xcode покажет граф объектов в памяти с их связями.
4. Фиолетовым отмечены объекты с циклическими ссылками (потенциальные утечки).

---

### Network Link Conditioner — имитация медленной сети

Встроенный в macOS инструмент для тестирования при плохой сети:

1. Скачай **Additional Tools for Xcode** с [developer.apple.com](https://developer.apple.com/download/all/) (раздел Downloads → More Downloads).
2. Установи **Hardware IO Tools** — появится **Network Link Conditioner** в System Preferences.
3. Включи и выбери пресет:

   | Пресет | Условия |
   |--------|---------|
   | **3G** | Мобильный интернет |
   | **Edge** | Очень медленная мобильная сеть |
   | **Very Bad Network** | Высокие потери, задержки |
   | **100% Loss** | Полная потеря пакетов |

4. Вся сеть Mac (и симулятора) пойдёт через эти условия.
5. В приложении проверь: курсы грузятся медленно → таймаут → кэш → сообщение об ошибке.

> ⚠️ **Не забудь выключить Network Link Conditioner после тестирования!** Иначе весь интернет на Mac будет работать через ограничения.

### Полезные функции симулятора

| Функция | Где найти | Зачем |
|---------|-----------|-------|
| Имитация встряхивания | **Device → Shake** | Тестирование жестов |
| Замедление анимаций | **Features → Toggle Slow Animations** | Визуальная проверка переходов |
| Скриншот | **File → Save Screen** | Для баг-репортов |
| Запись видео | **File → Record Screen** | Видео воспроизведения бага |
| Настройка статус-бара | **Features → Status Bar Override** | Красивые скриншоты (время, батарея, оператор) |
| Реальный размер | **Window → Physical Size** | Оценка UI на реальном масштабе |

### Установка на реальное устройство

Если нужно протестировать на реальном iPhone/iPad:

1. Выбери устройство в списке вместо симулятора.
2. Настрой подписание: таргет Runner → **Signing & Capabilities** → выбери свою команду.
3. При необходимости зарегистрируй устройство в Apple Developer.
4. Логи с реального устройства можно смотреть в **Window → Devices and Simulators** → выбери устройство → **Open Console**.
