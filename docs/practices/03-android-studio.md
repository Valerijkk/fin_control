# Практика: Android Studio — установка в эмуляторе и настройка логирования

**Одно приложение** FinControl. Задача: установить его в эмулятор (AVD), запустить и настроить **Logcat** — фильтр по пакету приложения, чтобы видеть логи при работе с экранами (Список, Обменник, Портфель).

## Цель

Установить приложение FinControl в эмуляторе Android Studio (AVD), запустить его и настроить просмотр логов (Logcat) по процессу приложения — чтобы при работе с экранами (Список, Обменник, Портфель) видеть логи в реальном времени.

## Ожидаемый результат

- Создан и запущен эмулятор (AVD). Приложение FinControl установлено в эмулятор (через `flutter run` или установку APK) и открывается с главного экрана.
- В Android Studio открыта панель **Logcat**; выбраны устройство и процесс приложения (пакет `com.yourname.fincontrol.fin_control` или как в `android/app/build.gradle.kts`).
- При переходах по экранам и действиях в приложении в Logcat появляются новые строки логов (Flutter/Dart и системные).

---

## Что понадобится

- Android Studio с установленным Android SDK
- Проект FinControl (Flutter) или собранный APK

## Шаг 1: Создание и запуск эмулятора (AVD)

1. Открой Android Studio (проект FinControl должен быть открыт из **корневой** папки, см. [00-getting-started.md](00-getting-started.md)).
2. В меню выбери **Tools → Device Manager** (или **AVD Manager**).
3. Нажми **Create Device** → выбери устройство (например Pixel 6) → **Next**.
4. Выбери системный образ с API 30 или выше (при необходимости нажми **Download** рядом с образом) → **Next** → **Finish**.
5. В списке устройств нажми кнопку **Run** (▶) напротив созданного AVD. Дождись загрузки эмулятора — появится окно с виртуальным устройством.

## Шаг 2: Установка FinControl в эмулятор

### Вариант A: из Flutter-проекта

Из корня проекта FinControl в терминале:

```bash
flutter run
```

Выберите запущенный эмулятор из списка. Приложение соберётся и установится.

### Вариант B: установка готового APK

1. Соберите APK: `flutter build apk --debug` (или `--release`).
2. APK будет в `build/app/outputs/flutter-apk/app-debug.apk` (или `app-release.apk`).
3. Перетащи APK в окно эмулятора или выполни:
   ```bash
   adb install -r build/app/outputs/flutter-apk/app-debug.apk
   ```
4. Запусти приложение на эмуляторе из лаунчера.

## Шаг 3: Настройка Logcat в Android Studio

1. Открой панель **Logcat**: меню **View → Tool Windows → Logcat** или вкладка **Logcat** в нижней части окна.
2. В выпадающем списке **устройств** выбери свой запущенный эмулятор (например `emulator-5554`).
3. В выпадающем списке **процессов** (часто подписан «Show only selected application» или по имени пакета) выбери пакет приложения FinControl: `com.yourname.fincontrol.fin_control` (точное значение смотри в `android/app/build.gradle.kts` — строка `applicationId`). Так в логах будут видны только сообщения от твоего приложения.
4. Уровень логирования при необходимости измени на **Verbose**, **Debug** или **Info**.
5. Если в коде используются теги (например `FinControl`), в поле поиска Logcat можно ввести этот тег для фильтрации.

## Шаг 4: Просмотр логов во время работы приложения

1. Запусти FinControl в эмуляторе.
2. Выполни сценарии: открытие экрана Обменник (сетевой запрос), Портфель, добавление расхода.
3. В Logcat должны появляться сообщения от Flutter/Dart и нативные логи (если они добавлены в коде).

## Проверка

- [ ] Эмулятор создан и запущен (окно эмулятора открыто).
- [ ] FinControl установлен и открывается из лаунчера эмулятора; приложение показывает главный экран с навигацией.
- [ ] В Logcat выбран ваш эмулятор и процесс приложения (пакет FinControl); при переходах по экранам (Список → Обменник → Портфель) в логах появляются новые строки.

## Траблшутинг

- **Эмулятор не появляется в списке при `flutter run`** — сначала запусти эмулятор из Android Studio (Tools → Device Manager → Run), затем выполни `flutter devices` и `flutter run`. [FAQ — Эмулятор не появляется в списке устройств](../FAQ.md#эмулятор-не-появляется-в-списке-устройств-при-flutter-run).
- **В списке процессов нет пакета приложения** — убедись, что приложение хотя бы раз было запущено на этом эмуляторе; проверь package name в `android/app/build.gradle.kts` ([FAQ — Где указан package name](../FAQ.md#где-в-проекте-указан-package-name-applicationid-приложения)).

## Что показать на экзамене / созвоне

1. Покажи запущенный эмулятор с открытым FinControl.
2. Открой **Logcat** в Android Studio, выбери устройство и процесс приложения (пакет `com.yourname.fincontrol.fin_control`).
3. В приложении перейди по экранам (Список → Обменник → Портфель) — в Logcat должны появляться новые строки (логи Flutter/системы).
4. Кратко скажи: «Создал AVD, установил приложение через Flutter run / APK. В Logcat отфильтровал по пакету приложения — при работе с экранами вижу логи.»

## Практические сценарии в Android Studio с FinControl

### Сценарий 1: Отслеживание логов при навигации

1. Запусти FinControl на эмуляторе.
2. Открой **Logcat**, в фильтре введи `[FinControl]`.
3. В приложении нажми **Начать** на приветственном экране.
4. **В Logcat увидишь:**
   - `[FinControl] Запуск приложения...`
   - `[FinControl] Данные загружены: N записей, M категорий`
   - `[FinControl] HomeScreen: initState — загрузка курсов и списка расходов`
5. Перейди на **Обменник** → в Logcat:
   - `[FinControl] ExchangeScreen: курсы загружены (exchangerate.host), 10 валют`
6. Сделай обмен → в Logcat:
   - `[FinControl] ExchangeScreen: обмен 1000,00 RUB → 10,50 USD (курс 0,01)`

### Сценарий 2: Инспекция базы данных через App Inspection

1. **View → Tool Windows → App Inspection**.
2. Выбери процесс FinControl, вкладка **Database Inspector**.
3. Открой таблицу `expenses` — увидишь все расходы/доходы.
4. Добавь расход в приложении (Кофе, 350 ₽, Еда).
5. В Database Inspector нажми **Refresh** — новая запись появится.
6. Выполни SQL-запрос:
   ```sql
   SELECT title, amount, category FROM expenses WHERE category = 'Еда';
   ```
7. **Что проверить:** запись «Кофе» присутствует, сумма 350, категория «Еда».

### Сценарий 3: Проверка утечек памяти через Memory Profiler

1. Открой **Profiler** → выбери процесс FinControl → вкладка **Memory**.
2. Запомни текущий уровень памяти (например 80 MB).
3. В приложении: Список → Обменник → Портфель → Акции → назад → Обменник → назад (10 раз).
4. В Profiler нажми **Force Garbage Collection** (иконка мусорного бака).
5. **Проверь:** вернулась ли память к исходному уровню (~80 MB)?
6. Если память растёт после GC — возможна утечка. Нажми **Dump Java Heap** для анализа.

### Сценарий 4: Анализ сетевого трафика через Network Profiler

1. В **Profiler** выбери вкладку **Network**.
2. В приложении открой **Обменник** → нажми **Обновить**.
3. На графике появится всплеск — это запрос к API курсов.
4. Кликни на запрос → увидишь:
   - URL: `https://api.exchangerate.host/latest?base=RUB&symbols=...`
   - Время выполнения (в мс)
   - Размер ответа (в байтах)
   - Статус: 200
5. **Сравни** с Charles/Proxyman — данные должны совпадать.

## Дополнительно: инструменты Android Studio для тестирования

### Layout Inspector — проверка UI в реальном времени

Layout Inspector — визуальный инструмент для инспекции иерархии виджетов работающего приложения.

1. Запусти приложение на эмуляторе.
2. **Tools → Layout Inspector** → выбери процесс FinControl.
3. Откроется окно с 3D-представлением иерархии виджетов.
4. Кликни на элемент — в панели справа увидишь его свойства (размеры, отступы, текст, цвет).

**Что проверять:**
- Правильность вложенности элементов (нет ли лишних контейнеров).
- Размеры и отступы (соответствуют ли дизайну).
- Видимость элементов (не перекрывают ли друг друга).
- Текст (обрезается ли длинный текст, корректна ли локализация).

### App Inspection — база данных и SharedPreferences

1. **View → Tool Windows → App Inspection**.
2. Выбери процесс FinControl и вкладку **Database Inspector**.
3. Увидишь все таблицы SQLite: `expenses`, `exchange_operations`, `portfolio_holdings`, `portfolio_transactions`, `price_alerts`, `limit_orders`, `savings_goals`.
4. Можно выполнять SQL-запросы в реальном времени:
   ```sql
   SELECT * FROM expenses ORDER BY date DESC LIMIT 10;
   ```
5. Проверь: добавь расход в приложении → обнови таблицу → запись должна появиться.

### Network Profiler — анализ сетевого трафика

1. **View → Tool Windows → Profiler** → выбери процесс.
2. Вкладка **Network** — график сетевой активности.
3. Открой экран Обменник → на графике появится всплеск (запрос к API курсов).
4. Кликни на запрос — увидишь метод, URL, заголовки, тело ответа, время выполнения.

**Отличие от Charles/Proxyman:** Network Profiler показывает трафик без необходимости настройки прокси и сертификатов, но менее удобен для модификации запросов.

### Memory Profiler — утечки памяти

1. В **Profiler** выбери вкладку **Memory**.
2. Походи по экранам приложения (Список → Обменник → Портфель → назад → снова).
3. Нажми **Force Garbage Collection** (иконка мусорного бака).
4. Если память не уменьшается после GC — возможна утечка.
5. **Dump Java Heap** → анализ объектов в памяти.

**Сценарий:** открой 10 раз экран Обменник, вернись назад → проверь, что старые экраны освобождают память.

### CPU Profiler — производительность

1. В **Profiler** выбери **CPU**.
2. Нажми **Record** → выполни действие в приложении (обмен валюты) → **Stop**.
3. Увидишь flame chart — какие методы заняли больше всего времени.
4. Ищи: длинные операции на main thread (могут вызвать зависание UI).

### Logcat: продвинутые фильтры

Помимо базового фильтра по пакету:

- **По уровню:** выбери **Error** — только ошибки и крэши.
- **По тегу:** в поле фильтра введи `tag:flutter` или `tag:FinControl`.
- **По содержимому:** введи текст, например `rates` — увидишь только логи, связанные с курсами.
- **Комбинированный:** `package:com.yourname.fincontrol.fin_control level:error` — ошибки только от приложения.
- **Regex:** включи regex-фильтр для сложных шаблонов (например `Exception|Error|FATAL`).

### Сохранение логов в файл

В Logcat: кнопка **Export to text file** (иконка дискеты) или через ADB:

```bash
adb logcat -d > logcat_$(date +%Y%m%d_%H%M%S).txt
```

Приложи файл логов к баг-репорту — это стандартная практика.

### Device File Explorer — файловая система устройства

1. **View → Tool Windows → Device File Explorer**.
2. Навигация по файловой системе устройства/эмулятора.
3. Путь к данным приложения: `/data/data/com.yourname.fincontrol.fin_control/` (доступно в debug-сборке).
4. Здесь можно найти: `databases/fin_control.db` (SQLite), `shared_prefs/` (SharedPreferences), кэш.
5. Правый клик → **Save As** — скачай файл на ПК для анализа.
