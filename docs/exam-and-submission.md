## 1. Что хотят в целом

- **Проект из коробки:** всё уже настроено; для метрик и Firebase достаточно вставить токены в **указанный файл** ([STUDENT_ENV.md](STUDENT_ENV.md) — единственное место для DSN Sentry и API Key AppMetrica) и для Firebase добавить конфиги по [инструкции](practices/00-firebase-setup.md).
- **Выполнить практики** по порядку из [practices-step-by-step.md](practices-step-by-step.md) и инструкциям в [practices/](practices/).
- **Показать результат**: на созвоне продемонстрировать, что всё работает, и кратко рассказать, что делали.
- **Подтвердить понимание**: не просто «нажал кнопку», а «настроил прокси на устройстве, установил сертификат Charles, открыл приложение — в Charles виден запрос к API курсов».

От вас **не** требуют идеальных скриншотов или отчётов на 10 страниц. Нужны: **выполненная практика + умение показать и объяснить**.

## 2. Как готовиться к сдаче

1. **Проходите практики по порядку** из [practices-step-by-step.md](practices-step-by-step.md): 00-getting-started → 01–09 → 00 Firebase setup → 10–15.
2. **Используйте пошаговые инструкции** в [practices/](practices/) — там расписано, что нажимать и что должно получиться. При ошибках — [FAQ.md](FAQ.md).
3. **Сверяйтесь с чек-листами** в [acceptance-criteria/](acceptance-criteria/) — по каждой практике файл с критериями приёмки (например [01-charles.md](acceptance-criteria/01-charles.md), [06-sentry.md](acceptance-criteria/06-sentry.md)); отметьте все пункты «Обязательно» перед сдачей.
4. **За день до созвона** проверьте по каждой сдаваемой практике:
   - приложение/Charles/Sentry/AppMetrica открываются и работают;
   - можете за 2–5 минут показать результат (запрос в Charles, логи в Logcat, событие в Sentry и т.д.);
   - можете одним-двумя предложениями объяснить, что делали и чему научились (см. таблицы ниже).
5. **Если сдаёте удалённо** — проверьте, что можете показать экран (Share Screen): эмулятор или телефон, Charles/Proxyman, браузер с Sentry/AppMetrica. Закройте лишние вкладки и приложения.

---

### Как обычно проходит созвон

- **Формат:** экзаменатор просит по очереди показать каждую сданную практику. На одну практику обычно отводят **2–5 минут**: вы показываете результат (экран приложения, Charles с запросом, Sentry с событием и т.д.) и кратко говорите, что делали.
- **Что показывать:** то, что описано в таблицах ниже в разделе «Что показать» — без длинных объяснений, по делу. Экзаменатор может задать уточняющий вопрос («Зачем нужен сертификат?», «Что такое DSN?») — ответы на такие вопросы см. в разделе [Типичные вопросы экзаменатора](#8-типичные-вопросы-экзаменатора).
- **Если что-то не работает в момент созвона:** не паникуйте. Можно попробовать перезапустить приложение или обновить страницу; если среда сломана — кратко опишите, что делали при выполнении практики. Экзаменатор может засчитать по описанию или договориться о повторной проверке.

## 4. Пошаговое выполнение: как не потеряться

Для каждой практики в папке [practices/](practices/) есть:

1. **Цель** — зачем это нужно.
2. **Шаги** — что делать по порядку.

Рекомендация: **выполняйте шаги подряд**, ставьте галочки в чек-листе. Если что-то не получилось — в разделе «Устранение неполадок» в практике, в [practices-step-by-step.md](practices-step-by-step.md) (блоки «Типичные ошибки») или в [FAQ.md](FAQ.md) часто есть подсказки.

## 5. Скриншоты и отчёты

- **Скриншоты** обычно не обязательны, если на созвоне вы показываете всё вживую. Если преподаватель просит «прислать отчёт» — достаточно 1–2 скрина по ключевым шагам (Charles с запросом, Sentry с событием, Logcat с логами и т.д.) и краткого списка: что сделали, что получилось.
- **Поэтапное выполнение** — это как раз шаги в практиках. На экзамене можно сказать: «Сначала создал AVD, потом установил приложение, затем открыл Logcat и отфильтровал по пакету» — и показать результат.

## 8. Типичные вопросы экзаменатора

Ниже — примеры вопросов, которые может задать экзаменатор, и краткие ориентиры для ответа. Вопросы и чек-листы **согласованы с [acceptance-criteria/](acceptance-criteria/)**: детальные критерии «что проверить» по каждой практике — в соответствующих файлах ([01-charles.md](acceptance-criteria/01-charles.md), [06-sentry.md](acceptance-criteria/06-sentry.md) и т.д.); сценарии «что показать» в этом разделе им соответствуют.

| Вопрос | О чём ответить кратко |
|--------|------------------------|
| **Что ты делал в этой практике?** | Перечисли ключевые шаги: например «Создал эмулятор в Android Studio, установил приложение через Flutter run, в Logcat отфильтровал по пакету приложения и смотрел логи при переходах по экранам». |
| **Зачем нужен Charles / Proxyman?** | Чтобы перехватывать и просматривать HTTP/HTTPS-трафик приложения (запросы к API, ответы). Нужно для отладки, проверки, что приложение обращается к правильным адресам и получает корректные данные. |
| **Почему нужен сертификат на устройстве?** | Чтобы расшифровать HTTPS: без доверенного сертификата Charles/Proxyman мы видим только факт соединения, но не содержимое запроса и ответа (оно зашифровано). |
| **Что такое DSN в Sentry?** | Это адрес (URL) проекта в Sentry, по которому приложение отправляет события и краши. Без правильного DSN события не попадут в твой проект. |
| **Зачем AppMetrica в приложении?** | Для аналитики: сессии, устройства, поведение пользователей. Позволяет понимать, как пользуются приложением, без доступа к устройствам пользователей. |
| **Что показать по Firebase?** | Что создан свой проект, добавлено приложение Android с правильным package name, в проекте лежит google-services.json, в main.dart вызывается Firebase.initializeApp(), и приложение запускается без ошибок. Для 10–15 — показать соответствующий модуль (Crashlytics, FCM и т.д.) в консоли и в приложении. |
| **Что делать, если на созвоне что-то не открылось?** | Спокойно повторить шаг (перезапустить приложение, обновить страницу в Sentry). Если среда сломана — кратко описать, что делал при выполнении практики и что показывал бы при рабочей среде; экзаменатор может засчитать по описанию или перенести проверку. |
| **Где в проекте вставляются токены Sentry и AppMetrica?** | Только в один файл — **lib/config/student_env.dart** (единственное место; проект из коробки, после подстановки всё работает). Firebase — конфиги по инструкции в practices/00-firebase-setup.md. |

### Продвинутые вопросы (для отличников)

| Вопрос | О чём ответить кратко |
|--------|------------------------|
| **Как подменить ответ API в Charles?** | Breakpoints: остановить ответ, изменить тело JSON (например, подставить другой курс валюты), отправить дальше. Или Map Local: подготовить файл с ответом и подставить его вместо реального запроса. |
| **Что такое Throttling и зачем он?** | Имитация медленной сети (3G, Edge). Позволяет проверить, как приложение ведёт себя при плохом интернете: показывает ли индикатор загрузки, использует ли кэш, не зависает ли UI. |
| **Что такое monkey-тестирование?** | Автоматическая генерация случайных действий (тапы, свайпы, повороты) через `adb shell monkey`. Проверяет устойчивость приложения к непредсказуемым действиям пользователя — ищет краши и ANR. |
| **Как найти утечку памяти?** | В Android Studio: Memory Profiler (Profiler → Memory), походить по экранам, вызвать GC — если память не уменьшается, возможна утечка. В Xcode: Memory Graph Debugger (Debug bar). |
| **Что такое breadcrumbs в Sentry?** | Цепочка действий пользователя перед ошибкой: какие экраны открывал, какие кнопки нажимал. Помогает воспроизвести баг без общения с пользователем. |
| **В чём разница между Crashlytics и Sentry?** | Оба отслеживают краши. Crashlytics — часть Firebase, тесно интегрирован с другими сервисами Google. Sentry — независимый сервис с более гибкой настройкой, breadcrumbs, performance monitoring. В реальных проектах часто используют оба. |
| **Зачем Remote Config?** | Изменение поведения приложения без обновления сборки: фичефлаги (вкл/выкл функцию), rollout (постепенная раскатка), A/B-тесты. Экономит время: не нужно пересобирать и заново загружать приложение в магазин. |
| **Как отслеживать регрессии производительности?** | Firebase Performance: кастомные трейсы (например время загрузки курсов), сравнение метрик между версиями приложения. Рост длительности при тех же условиях — регрессия. |

Ответы не нужно заучивать дословно — достаточно понимать суть и уметь коротко объяснить своими словами.
