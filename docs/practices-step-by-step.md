## Перед началом любых практик

### 1. Клонирование и первый запуск

**Что понадобится:** установленные Git и Flutter (проверка — команды `git --version` и `flutter --version` в терминале).

**Шаг 1. Открой терминал**
- **Windows:** PowerShell (Win + X → Windows PowerShell) или cmd. Можно также встроенный терминал в Cursor/VS Code (Ctrl+`).
- **Mac/Linux:** Terminal или встроенный терминал в IDE.

**Шаг 2. Перейди в папку для проектов**
- Пример: `cd Desktop` или `cd C:\Users\ИМЯ\Desktop`, или `cd ~/Projects`.
- Убедись, что ты не внутри другого репозитория (не должно быть скрытой папки `.git` в текущей директории, если хочешь положить fin_control рядом с другими проектами).

**Шаг 3. Клонируй репозиторий**
```bash
git clone https://github.com/Valerijkk/fin_control.git
cd fin_control
```
- Если `git clone` выдаёт ошибку «command not found» — установи Git с [git-scm.com](https://git-scm.com/).
- Если ошибка доступа/таймаут — проверь интернет и при необходимости настрой прокси для Git.

**Шаг 4. Проверь Flutter и установи зависимости**
```bash
flutter doctor
```
- Смотри вывод: для практик достаточно, чтобы был отмечен галочкой **Android** (или iOS на Mac). Красные крестики по Android Toolchain или Flutter нужно устранить (установить SDK, принять лицензии: `flutter doctor --android-licenses`).
```bash
flutter pub get
```
- Должно завершиться без ошибок. Это подтягивает пакеты из `pubspec.yaml`.

**Шаг 5. Запусти приложение**
```bash
flutter run
```
- Если устройств несколько — в терминале появится нумерованный список (например `[1] Pixel 9 API 34`, `[2] Chrome`). Введи номер нужного и Enter. Эмулятор нужно заранее запустить из Android Studio (Tools → Device Manager → Run).
- Первый запуск может занять 1–3 минуты (сборка Gradle, компиляция).
- **Что должно получиться:** на экране приложения приветственный экран с заголовком «FinControl» и кнопкой «Начать». После нажатия «Начать» открывается главный экран с нижней навигацией: **Список**, **Обменник**, **Акции**, **Портфель**, **Статистика**.

**Если не запускается:** типичные причины и решения — в разделе «Запуск и сборка» в [FAQ](FAQ.md). Кратко: проверь `flutter doctor`, что открыт корень проекта (не папка `android/`), выполни `flutter clean` и снова `flutter pub get`.

### 2. Как открывать проект в Android Studio (обязательно)

**Почему это важно:** в Android Studio нужно открывать **корневую папку проекта** — ту, где лежат файл `pubspec.yaml` и папки `lib/`, `android/`, `ios/`, `docs/`. **Не** открывай папку `android/` саму по себе: тогда IDE считает проект обычным Android-приложением, не видит Flutter/Dart, и появляются сотни ошибок Gradle и красные подчёркивания по всему коду.

**Пошагово:**
1. Запусти Android Studio. Если уже открыт другой проект — **File → Close Project** (вернёшься на экран приветствия).
2. На экране приветствия нажми **Open** (или **File → Open**).
3. В диалоге выбора папки укажи **папку fin_control** — именно ту, внутри которой лежат `pubspec.yaml`, `lib`, `android`, `ios`. Не выбирай папку `android` внутри неё.
4. Нажми **OK**. Дождись окончания индексации (внизу статус «Indexing…») и при необходимости **Gradle sync** (может предложить сам или через **File → Sync Project with Gradle Files**).
5. **File → Settings** (на Mac: **Android Studio → Settings**) → **Plugins**. Убедись, что установлены и включены **Flutter** и **Dart**. Если их нет — вкладка **Marketplace**, поиск «Flutter», установка, затем перезапуск Android Studio.

**Как проверить, что открыто правильно:** в леом дереве проекта (Project) должны быть видны папки `android`, `ios`, `lib`, `test`, файл `pubspec.yaml` в корне. При открытии папки `lib` файлы должны подсвечиваться как Dart (подсветка синтаксиса).

Подробнее и решение проблем: [ANDROID_STUDIO_LAUNCH.md](ANDROID_STUDIO_LAUNCH.md).

## Практика 03 — Android Studio: эмулятор и логи

**Цель:** установить FinControl в эмуляторе и видеть логи (Logcat) при работе с приложением.

### Шаг 1. Создать эмулятор (AVD)

1. В Android Studio: **Tools → Device Manager** (или иконка телефона с треугольником на панели).
2. Нажми **Create Device**.
3. Выбери устройство, например **Pixel 6** или **Pixel 9** → **Next**.
4. Выбери системный образ (рекомендуется **API 30** или выше). Если образа нет — нажми **Download** рядом с ним, дождись загрузки → **Next**.
5. Имя AVD можно не менять → **Finish**.
6. В списке устройств нажми **Run** (▶) напротив созданного эмулятора — должно открыться окно эмулятора с «телефоном».

**Что должно получиться:** эмулятор загрузился, на экране рабочий стол Android.

### Шаг 2. Установить FinControl в эмулятор

**Способ А (удобнее):** из корня проекта в терминале выполни:
```bash
flutter run
```
В списке устройств выбери номер твоего эмулятора (например `2` для Pixel 9 API 34). Дождись сборки — приложение установится и откроется.

**Способ Б:** собрать APK и поставить вручную:
```bash
flutter build apk --debug
adb install -r build/app/outputs/flutter-apk/app-debug.apk
```
(Эмулятор должен быть запущен.) Затем открой приложение «FinControl» в лаунчере эмулятора.

**Что должно получиться:** приложение FinControl открыто на эмуляторе, видна кнопка «Начать» или главный экран с навигацией.

### Шаг 3. Настроить Logcat

1. В Android Studio внизу открой вкладку **Logcat** (если не видна: **View → Tool Windows → Logcat**).
2. В выпадающем списке **устройств** выбери свой эмулятор (например «Pixel 9 API 34»).
3. В выпадающем списке **процессов** (часто написано «No Filters» или «Show only selected application») выбери пакет приложения: **com.yourname.fincontrol.fin_control**.  
   Если пакета нет в списке — запусти приложение на эмуляторе (открой его), затем снова открой список процессов — пакет должен появиться.
4. Уровень логов оставь **Verbose** или **Debug**.
5. В приложении походи по экранам: Список → Обменник → Портфель. В Logcat должны появляться новые строки (логи Flutter/системы).

**Что должно получиться:** в Logcat отфильтрованы логи только твоего приложения; при переходах по экранам список обновляется.

**Чек-лист перед сдачей:**
- [ ] Эмулятор создан и запущен.
- [ ] FinControl установлен и открывается.
- [ ] В Logcat выбран процесс `com.yourname.fincontrol.fin_control`, при действиях в приложении логи видны.

**Что сказать на экзамене:** «Создал AVD, установил приложение через Flutter run. В Logcat отфильтровал по пакету приложения — при переходах по экранам вижу логи.»

**Типичные ошибки:**
- **Пакета нет в списке процессов в Logcat** — убедись, что приложение именно запущено на выбранном эмуляторе (открой его на экране). Список процессов обновляется при выборе устройства; после запуска приложения пакет `com.yourname.fincontrol.fin_control` должен появиться.
- **Эмулятор не в списке при `flutter run`** — сначала запусти эмулятор из Android Studio (Device Manager → Run напротив AVD), подожди полной загрузки, затем снова выполни `flutter run`.

## Практика 01 — Charles: перехват трафика

**Цель:** увидеть в Charles HTTP/HTTPS-запросы приложения FinControl к API курсов валют.

### Шаг 1. Установить и запустить Charles

1. Скачай Charles с [charlesproxy.com](https://www.charlesproxy.com/) и установи.
2. Запусти Charles. При первом запуске может появиться запрос на установку сертификата — разреши (для расшифровки HTTPS на компьютере).

### Шаг 2. Узнать IP своего компьютера

- **Windows:** открой cmd или PowerShell, выполни `ipconfig`. Ищи «IPv4-адрес» в активном подключении (Wi‑Fi или Ethernet), например `192.168.1.105`.
- **Mac:** Системные настройки → Сеть → Wi‑Fi → Дополнительно → TCP/IP — адрес вида `192.168.1.x`.

Запомни этот IP — он понадобится для прокси на устройстве.

### Шаг 3. Настроить прокси на устройстве

**Если используешь эмулятор Android на том же компьютере:**

1. В эмуляторе: **Settings → Network & internet → Internet** (или **Wi‑Fi**).
2. Нажми на подключённую сеть (например «AndroidWifi») → **Изменить** / карандаш / **Advanced**.
3. **Прокси:** выбери «Вручную» (Manual).  
   **Имя хоста:** твой IP (например `192.168.1.105`).  
   **Порт:** `8888`.  
   Сохрани.
4. Эмулятор может переподключиться; если спрашивает «Подключиться к этой сети?» — да.

**Если используешь реальный телефон:** телефон и компьютер должны быть в одной Wi‑Fi сети. На телефоне: **Настройки → Wi‑Fi → долгое нажатие по сети → Изменить сеть / Дополнительно → Прокси: Вручную**, хост = IP компьютера, порт 8888.

### Шаг 4. Установить сертификат Charles на устройстве (для HTTPS)

1. В Charles: **Help → SSL Proxying → Install Charles Root Certificate on a Mobile Device or Remote Browser**.
2. В открывшемся окне будет адрес (например `chls.pro/ssl`) и порт. На **устройстве** открой **браузер** и перейди по этому адресу (можно ввести вручную: `chls.pro/ssl`).
3. Скачай и установи сертификат. На Android может потребоваться задать имя сертификата (например «Charles») и PIN/пароль разблокировки.
4. **Android 7+:** чтобы приложения доверяли Charles, сертификат нужно сделать «доверенным»: **Настройки → Безопасность → Шифрование и учётные данные → Установить сертификат → CA** (или «Доверенные учётные данные») и выбрать установленный сертификат Charles.

### Шаг 5. Включить SSL Proxying в Charles

1. **Proxy → SSL Proxying Settings**.
2. Включи **Enable SSL Proxying**.
3. В списке **Include** добавь: **Host** `*`, **Port** `443` (или конкретно `api.exchangerate.host` и `open.er-api.com`, порт 443).  
   Нажми **OK**.

### Шаг 6. Записать трафик и открыть приложение

1. В Charles убедись, что запись включена: **Proxy → Start Recording** (если пункт «Stop Recording» — значит уже пишет).
2. На устройстве/эмуляторе открой приложение **FinControl**.
3. Нажми «Начать», затем перейди на экран **Обменник** (иконка обмена внизу) или останься на главном — там подгружаются курсы.
4. В Charles в левом дереве должны появиться хосты. Найди запрос к **api.exchangerate.host** или **open.er-api.com** — кликни по нему. Справа откроется запрос/ответ; во вкладке **Response** должен быть JSON с полями курсов (rates, base и т.д.).

**Что должно получиться:** в Charles виден HTTPS-запрос к API курсов и расшифрованный ответ 200 с JSON.

**Если трафика нет:** смотри раздел «Charles / Proxyman не видит трафик» в [FAQ](FAQ.md).

**Типичные ошибки:**
- **В Charles запрос есть, но тело ответа «Unknown» или пустое** — HTTPS не расшифрован: на устройстве не установлен или не доверен сертификат Charles. Установи его заново и на Android 7+ добавь в «Доверенные учётные данные» (CA).
- **Трафика вообще нет** — проверь, что на устройстве в настройках Wi‑Fi/сети указан именно **ручной прокси** с IP твоего ПК и портом 8888; что Charles запущен и запись включена (Proxy → Start Recording); что в приложении открыт экран, где идёт загрузка курсов (Обменник или главный экран с карточкой курсов).

**Чек-лист перед сдачей:**
- [ ] Charles запущен, на устройстве прокси = IP компьютера, порт 8888.
- [ ] Сертификат Charles установлен и доверен на устройстве.
- [ ] SSL Proxying включён.
- [ ] В FinControl открыт экран с курсами (Обменник или главный).
- [ ] В Charles виден запрос к exchangerate.host или open.er-api.com, ответ 200, JSON.

## Практика 02 — Proxyman: перехват трафика

**Цель:** то же, что в Charles — перехватывать HTTPS-запросы FinControl к API курсов валют. Proxyman часто удобнее на Mac; есть версия и для Windows.

### Шаг 1. Установить и запустить Proxyman

1. Скачай Proxyman с [proxyman.io](https://proxyman.io/) под свою ОС (macOS или Windows).
2. Установи и запусти. При первом запуске разреши установку сертификатов (для расшифровки HTTPS на компьютере).

### Шаг 2. Узнать IP и порт прокси в Proxyman

1. В Proxyman открой настройки прокси: **Certificate → Proxy** (или **Tools → Set up Proxy**) — там будет указан **порт**. У Proxyman по умолчанию часто **9090** (в Charles — 8888). Запомни порт.
2. IP компьютера узнай так же, как для Charles: на Windows — `ipconfig` (IPv4-адрес), на Mac — Системные настройки → Сеть → Wi‑Fi → Дополнительно → TCP/IP.

### Шаг 3. Настроить прокси на устройстве

- **Эмулятор Android:** так же, как для Charles: Settings → Network & internet → Wi‑Fi → текущая сеть → Изменить → Прокси вручную. **Host:** IP компьютера. **Port:** 9090 (или тот, что показан в Proxyman).
- **Реальный телефон:** телефон и ПК в одной Wi‑Fi; в настройках Wi‑Fi на телефоне — прокси вручную, хост = IP ПК, порт = 9090.

### Шаг 4. Установить сертификат Proxyman на устройстве

1. В Proxyman найди раздел установки сертификата на мобильное устройство (например **Certificate → Install on Mobile Device** или аналог в меню).
2. На устройстве в **браузере** открой указанный в Proxyman URL (часто что-то вроде `proxy.man/ssl` или IP:порт).
3. Скачай и установи сертификат. На Android 7+ для доверия приложениям: **Настройки → Безопасность → Установить сертификат → CA** (или «Доверенные учётные данные») и выбери установленный сертификат Proxyman.

### Шаг 5. Включить перехват HTTPS и запись

1. В Proxyman убедись, что перехват SSL/TLS включён (часто по умолчанию уже включён для всех доменов). При необходимости добавь домены: `api.exchangerate.host`, `open.er-api.com`.
2. Включи запись трафика (кнопка Record/Capture, если есть).

### Шаг 6. Открыть FinControl и проверить трафик

1. На устройстве открой приложение FinControl → «Начать» → перейди на экран **Обменник** или останься на главном (карточка курсов).
2. В Proxyman в списке запросов должны появиться вызовы к **api.exchangerate.host** или **open.er-api.com**. Открой запрос — во вкладке ответа должен быть JSON с курсами (rates, base и т.д.).

**Чек-лист перед сдачей:** аналогично Charles — прокси на устройстве (IP + порт Proxyman), доверенный сертификат, в приложении открыты курсы, в Proxyman виден запрос к API и ответ с JSON.

**Что сказать на экзамене:** «Настроил Proxyman, на устройстве указал прокси (IP и порт из Proxyman), установил и доверил сертификат. В FinControl открыл экран с курсами — в Proxyman вижу HTTPS-запрос к API и расшифрованный ответ.»

## Практика 05 — ADB: команды и логи

**Цель:** освоить основные команды ADB на примере FinControl: проверка устройства, установка APK, просмотр логов, очистка данных, скриншот.

**Требования:** установленный Android SDK (обычно идёт с Android Studio) или отдельно platform-tools с `adb` в PATH. Эмулятор запущен или телефон подключён по USB с включённой отладкой по USB.

### Шаг 1. Проверить, что устройство видно

В терминале из **любой** папки выполни:
```bash
adb devices
```
- Должна появиться строка с идентификатором устройства (например `emulator-5554`) и статусом `device`. Если статус `unauthorized` — на телефоне подтверди разрешение отладки по USB.
- Если список пуст — запусти эмулятор или проверь кабель/драйверы USB.

### Шаг 2. Собрать и установить APK через ADB

Из **корня проекта** FinControl:
```bash
flutter build apk --debug
```
APK появится в `build/app/outputs/flutter-apk/app-debug.apk`. Установка (переустановка с сохранением данных):
```bash
adb install -r build/app/outputs/flutter-apk/app-debug.apk
```
- Должно вывести «Success». Открой приложение на устройстве из лаунчера.

### Шаг 3. Смотреть логи только приложения FinControl

Package name приложения: **com.yourname.fincontrol.fin_control** (из `android/app/build.gradle.kts`). Логи в реальном времени только этого процесса:
```bash
adb logcat --pid=$(adb shell pidof -s com.yourname.fincontrol.fin_control)
```
- Сначала запусти приложение на устройстве, затем выполни команду. При действиях в приложении в терминале будут появляться строки логов. Остановка: Ctrl+C.

### Шаг 4. Очистить данные приложения (как «Очистить данные» в настройках)

```bash
adb shell pm clear com.yourname.fincontrol.fin_control
```
- После этого при следующем запуске приложение будет как после первой установки (все расходы, настройки, кэш удалены).

### Шаг 5. Скриншот с устройства на ПК

```bash
adb exec-out screencap -p > screenshot.png
```
- Файл `screenshot.png` появится в текущей папке терминала.

**Чек-лист перед сдачей:**
- [ ] `adb devices` показывает устройство/эмулятор.
- [ ] Установил APK через `adb install -r ...`.
- [ ] Запустил приложение, посмотрел логи через `adb logcat --pid=...`.
- [ ] (По желанию) Выполнил `pm clear`, снова открыл приложение — данные сброшены; или сделал скриншот через `adb exec-out screencap`.

**Что сказать на экзамене:** «Установил APK через adb install, смотрю логи через adb logcat с фильтром по пакету приложения. При необходимости очищаю данные через pm clear или делаю скриншот через screencap.»

Подробнее все команды: [05-adb.md](practices/05-adb.md).

## Практика 06 — Sentry: события и краши

**Цель:** подключить Sentry к FinControl и увидеть тестовое событие в кабинете Sentry.

### Шаг 1. Создать проект в Sentry

1. Зайди на [sentry.io](https://sentry.io), зарегистрируйся или войди.
2. Создай организацию (если просят).
3. **Create Project** → платформа **Flutter** → имя проекта, например `fin-control` → **Create**.
4. После создания скопируй **DSN** — длинная строка вида `https://xxxx@xxxx.ingest.sentry.io/xxxx`. Она нужна для кода.

### Шаг 2. Вставить DSN в проект FinControl

1. В проекте открой файл **`lib/config/telemetry.dart`**.
2. Найди строку:
   ```dart
   const String sentryDsn = '';
   ```
3. Вставь свой DSN в кавычки:
   ```dart
   const String sentryDsn = 'https://xxxx@xxxx.ingest.sentry.io/xxxx';
   ```
4. Сохрани файл.

В проекте FinControl инициализация Sentry уже прописана в `lib/main.dart` — при непустом `sentryDsn` Sentry подхватится при запуске.

### Шаг 3. Отправить тестовое событие

1. Выполни `flutter run` и открой приложение на устройстве/эмуляторе.
2. Перейди в **Настройки:** на любом экране (Список, Обменник, Портфель и т.д.) в правом верхнем углу есть иконка **шестерёнки** — нажми её. Откроется экран «Настройки» с переключателем темы, кнопкой «Очистить все данные» и кнопкой **«Тест Sentry»**.
3. Нажми **«Тест Sentry»**. В приложении может ничего визуально не измениться — событие уходит в фоне. Если DSN был пустой до этого — после вставки DSN **обязательно перезапусти** приложение (`flutter run` заново или перезапуск с устройства), иначе кнопка могла не инициализировать отправку.
4. Открой в браузере кабинет Sentry → выбери свой проект → раздел **Issues** (или **Errors**). Через несколько секунд (иногда до 1–2 минут) должно появиться новое событие — тестовое исключение. Кликни по нему: откроется страница с **стек-трейсом**, информацией об устройстве, ОС, версии приложения и при необходимости breadcrumbs.

**Что должно получиться:** в Sentry в разделе Issues видно событие с тестовым исключением после нажатия «Тест Sentry».

**Типичные ошибки:**
- **Событие не появляется** — проверь, что DSN вставлен в `lib/config/telemetry.dart` без лишних пробелов и кавычки закрыты. Перезапусти приложение. Подожди 1–2 минуты и обнови страницу Issues в Sentry.
- **Кнопки «Тест Sentry» нет** — убедись, что открываешь именно экран «Настройки» (иконка шестерёнки в AppBar). В некоторых сборках кнопка может быть внизу экрана настроек.

**Чек-лист перед сдачей:**
- [ ] Проект Flutter создан в Sentry, DSN скопирован.
- [ ] В `lib/config/telemetry.dart` вставлен свой DSN.
- [ ] После нажатия «Тест Sentry» в приложении событие отображается в Sentry.

## Практика 07 — AppMetrica: сессии и устройства

**Цель:** подключить AppMetrica и увидеть сессию/устройство в отчётах.

### Шаг 1. Завести приложение в AppMetrica

1. Зайди на [appmetrica.io](https://appmetrica.io) (или appmetrica.yandex.ru), войди через Яндекс/Google.
2. Создай приложение (если ещё нет): укажи имя (например FinControl), платформу Android и/или iOS.
3. В настройках приложения найди **API Key** (число или строка) — скопируй его.

### Шаг 2. Вставить API Key в проект

1. Открой **`lib/config/telemetry.dart`**.
2. Найди:
   ```dart
   const String appMetricaApiKey = '';
   ```
3. Вставь свой ключ:
   ```dart
   const String appMetricaApiKey = 'ТВОЙ_API_KEY';
   ```
4. Сохрани. В `main.dart` инициализация AppMetrica уже есть — при непустом ключе она выполнится при старте.

### Шаг 3. Запустить приложение и проверить отчёт

1. Выполни `flutter run`, открой приложение, походи по экранам 1–2 минуты (Список, Обменник, Портфель, Настройки).
2. В кабинете AppMetrica ([appmetrica.io](https://appmetrica.io)) открой своё приложение → разделы **Отчёты**, **Устройства** или **Сессии**. Данные часто появляются с задержкой **5–15 минут** (особенно при первом запуске). Обнови страницу или подожди и зайди снова.

**Что должно получиться:** в AppMetrica в отчётах видны сессии и/или устройства после использования приложения.

**Типичные ошибки:**
- **Данных нет сразу** — задержка в AppMetrica нормальна. Подожди 10–15 минут, убедись, что приложение было открыто и им пользовались (несколько экранов), затем обнови отчёт.
- **API Key не принимается** — копируй ключ без пробелов в начале и конце. Ключ берётся из настройек приложения в кабинете AppMetrica (не из общего профиля).

**Чек-лист перед сдачей:**
- [ ] Приложение заведено в AppMetrica, API Key скопирован.
- [ ] В `lib/config/telemetry.dart` вставлен API Key.
- [ ] После запуска и использования приложения в отчётах AppMetrica видны данные.

## Практика 00 — Firebase: первый шаг (перед практиками 10–15)

Все практики по Firebase (Crashlytics, FCM, Analytics и т.д.) требуют **своего** проекта Firebase и конфигов в проекте.

### Шаг 1. Создать проект в Firebase

1. Зайди на [console.firebase.google.com](https://console.firebase.google.com), войди через Google.
2. **Создать проект** → имя, например `fin-control-practice` → при необходимости отключи/включи Analytics → дождись создания.

### Шаг 2. Узнать package name приложения Android

В проекте FinControl открой файл **`android/app/build.gradle.kts`**. Найди строку:
```kotlin
applicationId = "com.yourname.fincontrol.fin_control"
```
Это и есть **package name** — его нужно указать в Firebase при добавлении приложения Android. (Можно оставить как есть или заменить `yourname` на своё.)

### Шаг 3. Добавить приложение Android в Firebase

1. В обзоре проекта Firebase нажми **Добавить приложение** → иконка **Android**.
2. **Регистрация приложения** → в поле «Android package name» введи: `com.yourname.fincontrol.fin_control` (как в build.gradle.kts).
3. Никнейм и SHA-1 можно пропустить → **Зарегистрировать приложение**.
4. Скачай **google-services.json** и сохрани его на компьютер.
5. Перенеси этот файл в папку **`android/app/`** проекта FinControl (рядом с `build.gradle.kts`). В проводнике путь будет примерно: `fin_control/android/app/google-services.json`.

### Шаг 4. Подключить Firebase в коде (один раз)

1. В **`pubspec.yaml`** в блок `dependencies` добавь (если ещё нет):
   ```yaml
   firebase_core: ^3.0.0
   ```
   Выполни в терминале: `flutter pub get`.

2. В **Android:** в корневом `android/build.gradle.kts` в блок `buildscript` → `dependencies` добавь класс-путь Google Services (см. [документацию Flutter Firebase](https://firebase.flutter.dev/docs/installation)); в **`android/app/build.gradle.kts`** в конец файла добавь:
   ```kotlin
   apply(plugin = "com.google.gms.google-services")
   ```

3. В **`lib/main.dart`** в самом начале `main()` (до других инициализаций) добавь:
   ```dart
   import 'package:firebase_core/firebase_core.dart';

   Future<void> main() async {
     WidgetsFlutterBinding.ensureInitialized();
     await Firebase.initializeApp();
     // ... остальной код (Sentry, AppMetrica, runApp и т.д.)
   }
   ```

4. Запусти приложение: `flutter run`. Если в консоли нет ошибок про отсутствие конфигов — Firebase подключён. Дальше выполняй практики 10–15 по отдельным файлам в [docs/practices](practices/).

**Что должно получиться:** приложение запускается без ошибок, в папке `android/app/` лежит твой `google-services.json`.

Подробнее: [00-firebase-setup.md](practices/00-firebase-setup.md).

## Остальные практики (кратко)

По каждой из них в папке [docs/practices](practices/) есть отдельный файл с шагами. Ниже — самое важное и типичные подводные камни.

### Практика 04 — Xcode (только macOS)

- **Открытие проекта:** в Xcode открывай **не** корень репозитория и не `ios/Runner.xcodeproj`, а **`ios/Runner.xcworkspace`** (из-за CocoaPods/Flutter). Команда из терминала из корня проекта: `open ios/Runner.xcworkspace`.
- **Запуск:** в Xcode выбери схему **Runner**, в списке устройств — нужный симулятор (например iPhone 15), нажми **Run** (▶) или Cmd+R. Альтернатива: из корня проекта `flutter run` и выбрать iOS-симулятор в списке.
- **Логи:** во время работы приложения **View → Debug Area → Activate Console** (Cmd+Shift+C). В нижней панели (Console) видны вывод Flutter и нативного кода. При переходах по экранам в приложении в консоли появляются новые строки.
- Подробно: [04-xcode.md](practices/04-xcode.md).

### Практика 08 — TestFlight

- Требуется **Mac**, **Xcode** и **Apple Developer аккаунт** (платный). Сборка: Product → Archive; затем в Xcode Organizer — Distribute App → App Store Connect; в веб-интерфейсе App Store Connect настраивается TestFlight, добавляются тестеры. Подробно: [08-testflight.md](practices/08-testflight.md).

### Практика 09 — Android-дистрибуция

- Сборка подписанного AAB или APK (в `android/app/build.gradle.kts` настраивается signingConfig с keystore). Загрузка в Google Play Console (Internal testing) или в Firebase App Distribution. Подробно: [09-android-distribution.md](practices/09-android-distribution.md).

### Практики 10–15 — Firebase (Crashlytics, FCM, Analytics, Remote Config, Performance, In-App Messaging)

- **Обязательно сначала** выполнить [00-firebase-setup](practices/00-firebase-setup.md): свой проект в Firebase, добавление приложения Android (package name из `android/app/build.gradle.kts`), скачивание **google-services.json** и размещение в **`android/app/`**. В `lib/main.dart` в начале `main()` вызвать `await Firebase.initializeApp();`. Без этого все модули 10–15 будут падать с ошибкой «No Firebase App '[DEFAULT]' has been created».
- Дальше по номерам: подключать в `pubspec.yaml` и в коде нужные пакеты (firebase_crashlytics, firebase_messaging и т.д.) по инструкциям в файлах [10-firebase-crashlytics.md](practices/10-firebase-crashlytics.md) и далее.
